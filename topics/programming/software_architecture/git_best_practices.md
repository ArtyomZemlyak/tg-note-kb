# Лучшие практики Git

## Краткое описание

Лучшие практики Git представляют собой набор рекомендаций и подходов, направленных на обеспечение эффективной, безопасной и понятной работы с системой контроля версий. Следование этим практикам помогает избежать проблем, улучшает совместную работу и упрощает сопровождение проектов.

## Организация ветвления

### Модель ветвления Git Flow

- Основная ветка (main/master) - только стабильный код
- Ветка разработки (develop) - текущая разработка
- Функциональные ветки (feature/*) - разработка новых функций
- Ветки релизов (release/*) - подготовка релизов
- Ветки исправлений (hotfix/*) - срочные исправления

### Именование веток

- Используйте понятные имена: `feature/user-authentication`, `bugfix/login-error`
- Избегайте спецсимволов в именах веток
- Следуйте внутренним соглашениям команды

## Практики коммитов

### Сообщения коммитов

- Используйте информативные, краткие сообщения (до 50 символов для заголовка)
- Начинайте с глагола в повелительном наклонении: "Add", "Fix", "Update"
- Описывайте, что и зачем было изменено
- Используйте английский язык для сообщений коммитов в международных проектах

Пример хорошего сообщения коммита:
```
Fix user login validation

- Add proper email format validation
- Handle empty password error
- Update error message display
```

### Частота коммитов

- Пишите коммиты, соответствующие логически завершенным изменениям
- Не делайте слишком большие коммиты
- Не делайте слишком мелкие коммиты для крошечных изменений
- Используйте `git add -p` для частичного добавления изменений

## Безопасность и восстановление

### Регулярные пуши

- Регулярно отправляйте изменения в удаленный репозиторий
- Используйте `git push` для синхронизации работы
- Не храните важные изменения только локально

### Создание тегов

- Создавайте теги для релизов: `git tag -a v1.0.0 -m "Release version 1.0.0"`
- Используйте семантическое версионирование

## Работа с удаленными репозиториями

### Синхронизация

- Регулярно выполняйте `git fetch` для получения обновлений
- Используйте `git pull` или `git rebase` для синхронизации с основной веткой
- Проверяйте статус репозитория перед началом работы

### Работа с Pull Request'ами

- Используйте отдельные ветки для каждой функции
- Пишите информативные описания к Pull Request'ам
- Проходите код-ревью перед слиянием

## Инструменты и автоматизация

### Git Hooks

- Используйте pre-commit хуки для проверки кода
- Автоматизируйте форматирование кода
- Проверяйте качество кода перед коммитом

### .gitignore

- Используйте файл `.gitignore` для исключения временных файлов
- Не храните в репозитории чувствительные данные (ключи, пароли)
- Следуйте шаблонам для конкретных языков программирования

## Управление историей

### Интерактивное перебазирование

- Используйте `git rebase -i` для улучшения истории коммитов
- Комбинируйте, редактируйте и переупорядочивайте коммиты при необходимости
- Будьте осторожны при переписывании публичной истории

### Псевдонимы Git

Создайте полезные псевдонимы:
```bash
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.last 'log -1 HEAD'
git config --global alias.unstage 'reset HEAD --'
```

## Культура работы

- Регулярно синхронизируйте свою рабочую ветку с основной
- Тестируйте изменения перед пушем
- Уважайте чужие ветки и не переписывайте публичную историю
- Используйте код-ревью для повышения качества кода
